# 런치 하드닝 — 레이트리밋·에러 계측·브랜드 폰트

오너 승인으로 "지금 할 수 있는 건 다 제대로" 진행한 3건. 신규 npm 의존성 0개.

## 1. 레이트리밋 (우선순위 — 무료 엔드포인트가 실 LLM 비용을 태움)

- 신규 마이그레이션 `supabase/migrations/0002_rate_limits.sql` — 0001은 이미 프로덕션에 적용되어
  있어 손대지 않음, 순수 추가 파일. `rate_limits(key, window_start, count)` 고정폭 카운터 테이블 +
  `increment_rate_limit` 함수(원자적 증가 + 자기 키의 낡은 윈도우 삭제, opportunistic cleanup) +
  RLS deny-all(0001과 동일한 `drop policy if exists` 멱등 패턴 — 재확인 완료, 두 번 붙여넣어도 안전).
- `web/lib/rate-limit.ts` — 백엔드 2종(Supabase RPC / 인메모리 Map), `services.ts`의
  `useSupabase()`와 동일한 감지 패턴. **FAIL-OPEN**: 백엔드 에러 시 요청은 통과시키되
  `[RATE_LIMIT_BACKEND_ERROR]`로 로그. 고정폭 알고리즘, `enforceRateLimit(kind, key)` 진입점,
  키는 인증 시 `userId`·비인증 시 해시된 IP(HMAC-SHA256, `PII_HASH_KEY` 페퍼).
- 적용 대상 7개: 무료 티저 5/시간, 오라클 뽑기 30/일, 유료 리딩 생성 20/시간, 체크아웃 주문생성
  10/시간, 리퍼럴 적용 10/시간, 궁합방 생성 5/시간, 궁합방 참여 20/시간(친구 여럿이 짧은 시간에
  들어오는 정상 케이스라 생성보다 넉넉하게). 전부 team-lead 제안값 그대로, 변경 없음.
- 웹훅·크론 라우트(`payments/stripe/webhook`, `payments/reconcile`, `retention/sweep`)는 제외
  (자체 시크릿으로 보호됨).
- API 라우트(체크아웃·리퍼럴)는 실 HTTP 429 + `Retry-After` 헤더, RSC/서버액션 경로(티저·오라클·
  궁합방·리딩 결과 페이지들)는 429를 못 내므로 `RateLimitNotice` 컴포넌트로 인보이스 메시지 표시
  ("Whoa, the acorns need a breather — try again in a bit. 🐿️").
- 테스트: 리미터 유닛(윈도우 롤오버·양쪽 백엔드 동일 동작·fail-open·키 해싱) + 리퍼럴 라우트
  429 실증 테스트.

**⚠️ 중요 — 지금 당장 사실상 무방비**: `web/.env`에 이미 실 Supabase 자격증명이 있어서
`useSupabaseBackend()`가 true를 반환 → 레이트리밋은 "인메모리 폴백"이 아니라 무조건 Supabase RPC를
시도함. 0002 마이그레이션을 실행하기 전까지는 그 RPC가 테이블 부재로 매번 실패 → fail-open이라
서비스는 안 죽지만 **모든 요청이 무제한 통과**(로그만 남음). `NEEDS_FROM_OWNER.md`에 🔴로 명시.

## 2. 에러 계측 (외부 서비스는 오너 결정 대기 — 이번엔 로그+에러페이지만)

- `web/app/error.tsx`(세그먼트 경계) / `web/app/global-error.tsx`(루트 레이아웃 실패 시,
  자체 `<html>/<body>` 렌더링) — 브랜드 톤 영문 에러 메시지, 에러 노출 없이 재시도 버튼.
- `web/instrumentation.ts` — Next 15 `onRequestError` 훅, 요청당 1줄 grep 가능한 JSON 로그
  (route/method/digest/message). 실제 로직은 `reportError()` 함수 하나로 캡슐화 — 나중에 Sentry
  등 실 sink로 교체할 지점을 명확히 함.
- 결제 관련 파일 전수 감사(`payment.ts`/`payment-stripe.ts`/체크아웃·환불·웹훅 라우트/
  `reading-flow.ts`/`oracle-flow.ts`) — 진짜 침묵 catch 2건 발견해 수정(`oracle-flow.ts`의
  LLM 실패 catch, `payment-stripe.ts`의 `markOrderFailed` 정리 실패 catch) — 기존 마커 스타일
  (`[PAYMENT_CREDIT_FAILED]` 등)과 동일한 형식으로 로그만 추가, 동작 변경 없음. 나머지는 이미
  적절히 로깅되고 있어 손대지 않음.

## 3. 브랜드 폰트 (Nunito, 번들)

- Google Fonts 공식 저장소(OFL 라이선스)에서 Nunito 가변 폰트(`Nunito[wght].ttf`, 400~900 전
  웨이트 포함) + `OFL.txt` 다운로드 → `web/app/fonts/`. 정적 3-웨이트 파일은 저장소에 없어
  가변 폰트 1개로 전환(용량·구현 모두 더 간단, `next/font/local`의 `weight: '400 700'` 옵션으로
  필요한 범위만 사용).
- `app/layout.tsx`에서 `next/font/local`로 로드해 기존 `--font-sans` CSS 변수 값 자체를 교체
  (신규 변수 도입 아님) — 컴포넌트는 전부 `var(--font-sans)`/`font-sans`만 참조하고 있어서
  컴포넌트 파일 변경 0건으로 완료.
- OG 이미지 라우트(`app/s/[id]/opengraph-image.tsx`)는 satori 기본 폰트만 쓰고 `fonts` 옵션을
  안 넘기므로 이번 변경과 무관 — 실제로 무변경 확인.
